# VoIPネットワークの設計

VoIP（Voice over IP）は、テレフォニー市場において急速に成長しています。コンバージェンス（統合）のパラダイムは、私たちのコミュニケーションの方法を変え、コストを削減し、情報をやり取りする方法を強化しています。音声は、音声、ビデオ、プレゼンスを含む完全なマルチメディア・コミュニケーション時代の始まりに過ぎません。将来、私たちは仕事のために移動するのではなく、仕事そのものを人々の元へ運ぶようになるでしょう。なぜなら、その方がクリーンで、速く、安価だからです。VoIPはこの革命の一部に過ぎません。本章での私たちの課題は、VoIPネットワークを設計することです。そのために、セッションプロトコルやコーデックといった概念、そして回線数や帯域幅をどのように決定（ディメンジョニング）するかを理解する必要があります。

## 学習目標

本章を読み終えると、以下のことができるようになります。

- VoIPの利点を理解する
- AsteriskがどのようにVoIPを扱うかを説明する
- SIPおよびIAXチャネルの概念を説明する
- 特定のデータチャネルに最も適したプロトコルを選択する
- 特定のデータチャネルに最も適したコーデックを選択する
- 必要なチャネル数を決定する
- 必要な帯域幅を計算する

## VoIPの利点

なぜVoIPに関心を持つべきなのでしょうか？VoIPは企業と個人の双方に利点をもたらします。コスト削減はその一つですが、環境によっては、VoIPはコンピュータシステムの統合を簡素化します。いくつかの利点を以下に詳述します。

### コンバージェンス

VoIPの最大の利点は、データネットワークと音声ネットワークを統合してコストを削減すること（コンバージェンス）です。しかし、音声通話の分単位のコストを分析するだけでは、VoIP導入の正当化には不十分かもしれません。電話会社が販売する通話料金は急速に安くなっており、VoIP導入前に考慮すべき点です。

### インフラストラクチャのコスト

単一のネットワークインフラストラクチャを使用することで、追加、削除、変更に関連するコストが削減されます。IPが普及するにつれ、携帯電話、PDA、組み込みシステム、ラップトップなど、多くの新しいデバイスにVoIP関連技術がもたらされました。

### オープンスタンダード

最後に、VoIPが構築されているオープンスタンダードは、異なるベンダーから自由に選択できる自由を提供します。この一つの利点により、顧客は通信事業者（TELCO）やPBXメーカーの従属物ではなく、王様になることができます。

### コンピュータテレフォニー統合（CTI）

テレフォニーはコンピューティングよりもはるかに古い歴史を持っています。従来のPBXは回線交換ベースであり、通常、監視用のコンピュータが1台ある程度です。VoIPでは、テレフォニーは最初からコンピュータの標準に基づいて作成されています。これにより、コンピュータテレフォニー・アプリケーションの使用が、古いモデルよりも安価で容易になります。Asteriskをベースにしたテレフォニー・アプリケーションのリストを素早く作成できます。IVR、ACD、CTI、ダイヤラー、画面ポップアップなどのアプリケーションを、従来のPBXに比べてわずかな時間で開発できます。

## AsteriskのVoIPアーキテクチャ

Asteriskのアーキテクチャを以下に示します。AsteriskはすべてのVoIPプロトコルをチャネルとして扱います。どのコーデックやプロトコルでも使用できます。ここで学ぶべき概念は、Asteriskがあらゆる種類のチャネルを他のあらゆるチャネルにブリッジできるということです。したがって、SIPやIAXのようなシグナリングプロトコルを相互に変換したり、異なるコーデック間で変換したりすることさえ可能です。例えば、ローカルエリアネットワーク内のSIP電話からG.711コーデックを使用して、VoIPプロバイダーへのSIPトランクへG.729コーデックを使用して通話を変換できます。次章では、SIPとIAXのアーキテクチャの詳細を説明します。H.323のサポート（chan_ooh323アドオン経由）は利用可能ですが、ますます稀になっています。現代の導入環境ではSIP/PJSIPが標準です。

## VoIPプロトコルとネットワークスタック

VoIPは、連携して動作する一連の異なるプロトコルを使用します。これらを7層のOSI参照モデルに当てはめたくなる誘惑に駆られます。多くの古い図では、まさにそれを行っており、SIPやH.323を「セッション」層に、コーデックを「プレゼンテーション」層に配置しています。そのマッピングは常に議論の的となってきました。SIPを標準化しているIETFはOSIモデルを使用しておらず、より古い4層のTCP/IP（DoD）モデルに従っており、RFC 3261では**SIPをアプリケーション層プロトコル**と定義しています。メディアも同様のパターンに従い、RTPとコーデックはアプリケーション層のペイロードに存在し、トランスポート層のUDP上で運ばれます。以下の表は、主要なVoIPプロトコルをIETFが実際に使用しているTCP/IPモデルにマッピングしたもので、大まかなOSI相当層は参考としてのみ示しています。

| TCP/IP (IETF) 層 | プロトコル | 大まかなOSI相当層 |
|---|---|---|
| アプリケーション | SIP, H.323, MGCP, IAX2シグナリング; RTP/RTCP; コーデック (G.711, G.729, Opus…) | アプリケーション / プレゼンテーション / セッション |
| トランスポート | UDP, TCP | トランスポート |
| インターネット | IP (DiffServなどのQoSを含む) | ネットワーク |
| リンク | Ethernet, PPP, Frame Relay… | データリンク / 物理 |

DiffServのようなQoSメカニズムはIP層で動作し、音声パケットを優先して通話品質を向上させます。いくつかのプロトコルの詳細：

- **SIP**は、シグナリングを運ぶためにポート5060（TLSは5061）でUDPまたはTCPを使用します。音声は、設定可能なUDPポート範囲（Asteriskの出荷時の`rtp.conf`サンプルでは10000から20000を使用）でRTPによって個別に運ばれ、G.711のようなコーデックでエンコードされます。
- **H.323**は、TCP上で呼制御シグナリング（ポート1720でのH.225呼制御）を運び、H.225 RASチャネルはポート1719でUDPを使用します。RTPが音声を転送します。
- **IAX2**は特殊で、シグナリングとメディアの両方を単一のUDPポート（4569）で多重化するため、NATやファイアウォールの通過が簡素化されます。

> **[第2版注記]** 初版の図では、SIP/H.323をOSIの*セッション*層に、コーデックを*プレゼンテーション*層にマッピングしていました。そのマッピングは長年の論争の種でした。IETFはTCP/IPモデルを使用しており、その中でSIPはアプリケーション層プロトコルであるため、図は上記の表に置き換えられました。視覚的な図が必要な場合は、TCP/IPスタックに基づいた図の再作成を依頼してください。

## プロトコルの選択方法

多くのプロトコルがある中で、ネットワークに最適なものをどのように選択すればよいでしょうか？本節では、各プロトコルの利点と欠点を強調します。

### SIP - Session Initiated Protocol

SIPはIETF（Internet Engineering Task Force）のオープンスタンダードであり、主にRFC 3261で定義されています。ほとんどの現代のVoIPプロバイダーはSIPを使用しており、実際、最も人気のあるVoIP標準になりつつあります。SIPの強みは、IETFベースの標準であることです。SIPは、古いH.323と比較して軽量です。SIPの主な弱点はNATトラバーサルであり、これはほとんどのSIP VoIPプロバイダーにとって課題です。IETFは課金を念頭に置いてSIPを作成したのではなく、ピア間のオープンなコミュニケーションのために作成しました。課金は通常、VoIPプロバイダーにとっての懸念事項です。

### IAX – Inter Asterisk eXchange

IAXは、元々Digium（現在はSangoma）によって開発されたオープンプロトコルです。IAXは、シグナリングとメディアを同じUDPポート（4569）で転送するため、オールインワンのプロトコルです。Mark Spencerは、帯域幅を削減するためのバイナリプロトコルとしてIAXを開発しました。IAXの主な強みは、帯域幅使用量の削減（RTPを使用しない）です。また、単一のUDPポート（4569）しか使用しないため、NATやファイアウォールの通過が非常に容易です。もし従来のPBXメーカーがIAXを作成していたら、おそらく「アイスクリーム以来の最高の発明」として売り込んでいたでしょう。状況によっては、トランクモードのIAXは音声帯域幅の使用量を3分の1に削減できます。IAX2（バージョン2）は、Asterisk 22でも`chan_iax2`モジュール経由で提供されており、Asterisk間トランクとして依然として有用ですが、レガシーと見なされています。新規導入にはSIP/PJSIPが推奨されます。IAX2は[RFC 5456](https://www.rfc-editor.org/rfc/rfc5456)（情報提供）で規定されています。

### MGCP – Media Gateway Control Protocol

MGCPは、H.323、SIP、IAXと組み合わせて使用されるプロトコルです。その最大の利点はスケーラビリティです。ゲートウェイではなくコールエージェントで設定されるため、設定プロセスが簡素化され、集中管理が可能になります。しかし、Asteriskの実装は完全ではなく、あまり使用されていないようです。

### H.323

H.323はVoIPで広く使用されています。これは最初のVoIPプロトコルの一つであり、ゲートウェイベースの古いVoIPインフラストラクチャを接続するために不可欠です。H.323はゲートウェイ市場では依然として標準ですが、市場はゆっくりとSIPに移行しています。H.323の強みには、市場での広範な採用と成熟度が含まれます。H.323の弱点は、実装の複雑さと標準化団体に関連するコストです。

### プロトコル比較表

以下の表は、セッションプロトコル間の違いをまとめたものです。

| プロトコル | 標準化団体 | 用途 |
|----------|---------------|----------|
| SIP | IETF標準 | SIP電話; SIPサービスプロバイダーへの接続 |
| IAX2 | RFC 5456 (情報提供) | Asterisk間トランク; IAX2電話; IAXサービスプロバイダー |
| H.323 | ITU標準 | H.323電話およびゲートウェイ (外部ゲートキーパーを使用可能、自身は不可) |
| MGCP | IETF/ITU | MGCP電話 (プロバイダー/ゲートウェイのサポートなし) |
| SCCP | Cisco独自 | Cisco電話 |

> **[第2版注記]** 現在のAsterisk 22のモジュール名とサポート状況を反映するために、この表を更新することを検討してください。

## Peers、Users、Friends

SIPおよびIAXクライアントには3種類あります。一つ目は「user」です。UserはAsteriskサーバーに発信できますが、サーバーからの着信を受けることはできません。二つ目は「peer」です。Peerに対して発信できますが、彼らからの着信を受けることはできません。通常、サーバーやデバイスは両方の概念を同時に必要とします。「friend」は「user」+「peer」のショートカットです。電話機は発信と着信の両方が必要であるため、通常はこのカテゴリに分類されます。

![Asteriskの観点からのPeer、User、Friend：「user」はAsteriskへ発信し、「peer」はAsteriskからの着信を受け、「friend」はその両方を行います。](../images/06-voip-network-fig03.png)

> **[第2版注記]** Peer/User/Friendの区別は`chan_sip`（`sip.conf`）の概念です。PJSIP（`pjsip.conf`）では、これは着信と発信の両方を処理する**endpoint**オブジェクトに置き換えられています。Asterisk 22では、Peer/Userの分離は適用されなくなりました。

## コーデックとコーデック変換

音声信号をアナログ波からデジタル信号に変換するためにコーデックを使用します。コーデックは、音質、圧縮率、帯域幅、計算要件などの点で異なります。サービス、電話機、ゲートウェイは通常、これらの側面のいくつかをサポートしています。コーデックG.729は非常に人気があります。これは標準のAsterisk 22ビルドには含まれておらず、Digium（現在はSangoma）からダウンロードする外部アドオンモジュール（`codec_g729`）として提供されています。Asteriskの`menuselect`ソースには`support_level=external`と共に記載されており、「Digiumからg729aコーデックをダウンロードしてください。このコーデックにはライセンスの購入が必要です」と明記されています。言い換えれば、G.729を合法的に使用するには、チャネルごとのライセンス購入が必要です。（オープンソースの代替品である`bcg729`も存在します。）

![パルス符号変調（PCM）：4000 Hzのアナログ信号を1秒間に8000回サンプリングし（ナイキストの定理）、64 Kbpsのデジタルビットストリームに符号化します。](../images/06-voip-network-fig04.png)

Asterisk 22は（他にもありますが）以下のコーデックをサポートしています。

- GSM: 13 Kbps
- iLBC: 13.3 Kbps
- ITU G.711 (ulaw/alaw): 64 Kbps — 標準的なPSTN品質; 北米ではulaw、欧州やラテンアメリカではalawが一般的
- ITU G.722: 64 Kbps — ワイドバンド（HD音声）、G.711と同じ帯域幅で良好な品質
- ITU G.723.1: 5.3/6.3 Kbps
- ITU G.726: 16/24/32/40 Kbps
- ITU G.729: 8 Kbps — Digium/Sangomaからダウンロードする外部`codec_g729`バイナリモジュール（`support_level=external`; 使用にはライセンス購入が必要）
- Speex: 2.15から44.2 Kbps
- LPC10: 2.4 Kbps
- **Opus**: 6–510 Kbps、可変 — 現代のワイドバンド/フルバンドコーデック; 優れた品質とパケット損失耐性; Digium/Sangomaからダウンロードする外部`codec_opus`バイナリモジュールとして提供（`support_level=external`; G.729と異なりライセンス購入の記載なし）; WebRTCおよび現代のSIPエンドポイントに推奨。（GitHubにはオープンソースのビルド代替品も存在します。）

さらに、Asteriskはコーデック間の変換を許可しています。g723のようにパススルーモードでのみサポートされる場合など、変換が不可能なケースもあります。あるコーデックから別のコーデックへの変換は、CPUの多くのリソースを消費します。したがって、可能な限り避けてください。

## コーデックの選択方法

コーデックの選択は、以下のような複数のオプションに依存します。

- 音質
- ライセンスコスト
- CPU処理消費量
- 帯域幅要件
- パケット損失隠蔽
- Asteriskおよび電話デバイスでの利用可能性

以下の表は、最も人気のあるコーデックを比較したものです。これらのコーデックの品質は「トール（通話）」品質、つまりPSTNと同等と見なされています。

| コーデック | G.711 | G.729A | iLBC | GSM 06.10 |
|---|---|---|---|---|
| サンプル間隔 | 20 ms | 30 ms | 30 ms | RTE/LTP |
| 帯域幅 (Kbps) | 64 | 8 | 13.33 | 13 |
| コスト (チャネルあたり) | 無料 | ~USD 10.00 | 無料 | 無料 |
| フレーム消失耐性¹ | なし | 3% | 5% | 3% |
| 複雑度 (MIPS)² | ~0.35 | ~13 | ~18 | ~5 |

¹ パケット損失耐性とは、MOSが特定のコーデックのピーク品質から約0.5低下する割合を指します。

² 複雑度とは、Texas Instruments DSP (TMS320C54x) のリファレンス設計を使用してコーデックを符号化および復号化するために費やされる、1秒あたりの命令数（百万単位）を指します。プロセッサ周波数とMIPSの間には直接的な関係がありますが、多様なハードウェアプラットフォーム間で正確な関係を導き出すことは不可能です。この表は比較のためにのみ使用してください。

> **[第2版注記]** 第2版に向けてこの比較表を再構築してください：Opus（無料`codec_opus`）とG.722（ワイドバンド、64 Kbps）を追加し、G.711 ulaw/alawをPSTNのベースラインとして維持してください。G.729については、ライセンスの現実を注記してください。Sangomaの`codec_g729`はダウンロードは無料ですが、チャネルごとのライセンスが必要です。オープンソースの`bcg729`が代替手段です。

**Asterisk 22向けのコーデック推奨事項:**

- **G.711 (ulaw/alaw):** PSTNトランクおよび最大限の相互運用性に使用。Asterisk内でのトランスコーディングコストはゼロ。
- **G.729:** 低帯域幅のWANトランクに有用。Sangomaの`codec_g729`モジュールはダウンロードは無料だが、使用にはチャネルごとのライセンス購入が必要。
- **G.722:** LAN/内部内線でのワイドバンド（HD音声）に適した選択肢。G.711と同じ帯域幅でより高品質。
- **Opus:** 現代のエンドポイント、WebRTCクライアント、およびエンドポイントがサポートするあらゆる導入環境に推奨。適応型ビットレート、優れたパケット損失耐性、Sangomaの`codec_opus`バイナリモジュール経由で自由に入手可能。

## プロトコルヘッダーによるオーバーヘッド

コーデックの帯域幅消費はわずかですが、Ethernet、IP、UDP、RTPなどのプロトコルヘッダーによるオーバーヘッドを考慮する必要があります。そのため、帯域幅は使用されるヘッダーに依存すると言えます。Ethernetネットワークの場合、PPPヘッダーはEthernetヘッダーよりも短いため、PPPネットワークよりも帯域幅要件が高くなります。いくつかの例を見てみましょう：Ethernet宛先 G.729符号化 (20) UDPヘッダー (8) Ethernetタイプ (2) Ethernetソース IPヘッダー (20) RTPヘッダー (12) 音声ペイロード チェックサム (4) アドレス (6) アドレス (6) Ethernet コーデック g.711 (64 Kbps)

![Ethernet上の単一のg.729音声パケット：20バイトのペイロードが58バイトのEthernet、IP、UDP、RTPヘッダーでラップされています — g.729の会話は31.2 Kbpsを消費します。](../images/06-voip-network-fig05.png)

- Ethernet (Ethernet+IP+UDP+RTP+G.711) = 95.2 Kbps
- PPP (PPP+IP+UDP+RTP+G.711) = 82.4 Kbps
- Frame-Relay (FR+IP+UDP+RTP+G.711) = 82.8 Kbps

コーデック G.729 (8 Kbps)

- Ethernet (Ethernet+IP+UDP+RTP+G.729) = 31.2 Kbps
- PPP (PPP+IP+UDP+RTP+G.729) = 26.4 Kbps
- Frame-Relay (FR+IP+UDP+RTP+G.729) = 26.8 Kbps

## トラフィックエンジニアリング

VoIPネットワーク設計における主要な課題は、リモートオフィスやサービスプロバイダーなどの特定の宛先に対する回線数と必要な帯域幅を決定することです。Asteriskの同時通話数（Asteriskのディメンジョニングにおける主要なパラメータ）を決定することも重要です。

### 簡略化

最も基本的で広く使用されている簡略化は、ユーザータイプごとに通話数を推定することです。例：

- ビジネスPBX（内線5台につき同時通話1）
- 住宅ユーザー（ユーザー16人につき同時通話1）

例 #1 本社には120の内線があり、2つの支店があります。1つ目は30の内線、2つ目は15の内線です。私たちの目的は、本社のE1トランク数とFrame-Relayネットワークに必要な帯域幅を決定することです。

### Erlang B法

1.a VoIP同時通話数 簡略化が最善のアプローチではない場合があります。過去のデータがある場合は、より科学的なアプローチを採用できます。私たちは、2都市間のトランクグループ内の回線を計算する公式を開発したAgner Karup Erlang（コペンハーゲン電話会社、1909年）の研究を使用します。Erlangは、通常テレコムで見られるトラフィック測定単位です。これは1時間あたりのトラフィック量を表すために使用されます。例：1時間に20回の通話があり、平均5分の会話。Erlang数は以下のように計算できます：1時間あたりの通話分：20 x 5 = 100分 1時間あたりのトラフィック時間：100/60 = 1.66 Erlangs。これらの測定値は通話ログから判断でき、それを使用してネットワークを設計し、必要な回線数を計算できます。回線数がわかれば、帯域幅要件を計算できます。Erlang Bは、トランクグループ内の回線数を計算するために最も一般的に使用される方法です。これは、通話がランダムに到着し（ポアソン分布）、ブロックされた通話は即座にクリアされると仮定します。この方法では、通話ログから取得できる、または以下の簡略化によって得られるBusy Hour Traffic (BHT) を知る必要があります：BHT = 1日の通話分の17%。

![Erlang B計算結果：1%のブロッキングで5 Erlangsは11回線（本社から支店#1）を必要とし、1%のブロッキングで2.83 Erlangsは8回線（本社から支店#2）を必要とします。](../images/06-voip-network-fig07.png)

もう一つの重要な変数は、回線不足による通話ブロックの確率を定義するGrade of Service (GoS) です。このパラメータは任意に設定でき、通常は0.05（5%の通話損失）または0.01（1%の通話損失）です。例 #1：5.10.1と同じ例を使用して、トラフィックパターンに関するデータを提供します。通話ログから、以下のデータを発見しました：通話ログからのデータ（通話分とBHT）：

GoS=0.01とします

- 本社から支店#1 - BHT=300分/60 = 5 Erlangs
- 本社から支店#2 – BHT=170分/60 = 2.83 Erlangs

## VoIPに必要な帯域幅の削減

VoIP通話に必要な帯域幅を削減するために、3つの方法を使用できます：

- RTPヘッダー圧縮
- IAXトランク
- VoIPペイロード

### RTPヘッダー圧縮

Frame-RelayおよびPPPネットワークでは、RTPヘッダー圧縮を使用できます。RTPヘッダー圧縮はRFC 2508で定義されています。これはいくつかのルーターで利用可能なIETF標準です。ただし、このリソースを利用するために異なる機能セットを必要とするルーターもあるため、注意してください。RTPヘッダー圧縮を使用する影響は素晴らしく、例における音声会話あたりの必要な帯域幅を26.8 Kbpsから11.2 Kbpsに削減します。これは58.2%の削減です！

### IAX2トランクモード

2つのAsteriskサーバーを接続している場合、トランクモードでIAX2プロトコルを使用できます。この革新的な技術は特別なルーターを必要とせず、あらゆる種類のデータリンクに適用できます。

![Ethernet上のIAX2トランクモード：1つのg.729通話は完全なヘッダースタック（31.2 Kbps）を必要としますが、2番目の通話はそれらのヘッダーを共有し、小さなIAX2ミニフレームのみを追加するため、追加通話あたり平均約9.6 Kbpsの追加帯域幅となります。](../images/06-voip-network-fig08.png)

IAX2トランクモードは、2番目以降の通話から同じヘッダーを再利用します。PPPリンクでg729を使用する場合、最初の通話は30 Kbpsの帯域幅を消費しますが、2番目の通話は最初の通話と同じヘッダーを使用するため、追加通話に必要な帯域幅は9.6 Kbpsに削減されます。トランクモードで必要な帯域幅は次のように計算できます：支店#1（11通話）帯域幅 = 31.2 + (11-1)* 9.6 Kbps = 127.2 Kbps 支店#2（8通話）帯域幅 = 31.2 + (8-1)* 9.6 Kbps = 98.4 Kbps 最初の通話は31.2 Kbpsを使用し、次は9.6 Kbpsを使用します。

### 音声ペイロードの増加

この方法は、インターネット経由でVoIPゲートウェイを使用する場合に非常に一般的です。より大きなペイロードを使用すると、帯域幅削減と引き換えにレイテンシを犠牲にすることになります。allow命令でコーデックにフレームサイズを追加することで、RTPパケット化を変更できます。

![音声ペイロードの増加：60バイトのg.729ペイロードを1つのパケットにパックする（20バイトの代わりに）ことで、58バイトのヘッダーをより多くの音声に分散させ、レイテンシの増加と引き換えに帯域幅を1通話あたり約16.05 Kbpsに低下させます。](../images/06-voip-network-fig09.png)

例：

```
allow=ulaw:30
```

許可される値は以下の通りです：名前 最小 最大 デフォルト 増分 g723 gsm ulaw alaw g726 ADPCM SLIN lpc10 g729 speex ilbc

## まとめ

本章では、Asteriskがチャネルを使用してVoIPを扱うことを学びました。SIP（Asterisk 22では`chan_pjsip`経由）とIAX2をサポートしています。H.323はコミュニティの`ooh323`アドオンを通じてのみ利用可能であり、古いMGCPおよびSCCP（Skinny）チャネルは標準のAsterisk 22ビルドの一部ではなくなりました。シグナリングプロトコルとVoIPチャネル用のコーデックの比較と選択方法を学びました。IAX2は帯域幅効率が高く、NATを容易に通過できます。SIP/PJSIPはサードパーティの電話機やゲートウェイベンダーによって最もサポートされているプロトコルであり、Asterisk 22における唯一のSIPチャネルドライバーです。H.323プロトコルは最も古いものであり、レガシーVoIPインフラストラクチャへの接続に使用すべきです。5.11節では、VoIPネットワークの設計とディメンジョニング方法を学びました。
