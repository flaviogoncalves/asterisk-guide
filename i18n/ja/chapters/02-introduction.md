# Asterisk PBX 入門

FreePBX や Issabel といった、すぐに実行可能なディストリビューションの普及が近年進んでいます。本書では、これらのディストリビューションを理解するための基盤となる、古典的な Asterisk について解説します。Asterisk PBX は、一般的な PC を強力なマルチプロトコル PBX に変貌させることができるオープンソースソフトウェアです。本章では、この新しい技術の可能性とその基本的なアーキテクチャについて学びます。

## 学習目標

本章を読み終えることで、以下のことができるようになります。

- Asterisk とは何か、何ができるのかを説明する。
- Digium™ とその継承者である Sangoma の役割を説明する。
- Asterisk の基本アーキテクチャとそのコンポーネントを認識する。
- いくつかの利用シナリオを指摘する。
- 情報やヘルプの入手先を特定する。

## Asterisk とは

Asterisk はオープンソースの PBX ソフトウェアであり、適切なインターフェースを備えた PC ハードウェアにインストールすることで、家庭用ユーザー、企業、VoIP サービスプロバイダー、電話会社向けのフル機能 PBX として利用できます。Asterisk はオープンソースコミュニティであると同時に、Sangoma Technologies（2018年に Digium を買収）がスポンサーとなっているプロジェクトでもあります。Asterisk は、ニーズに合わせて自由に利用および改変できます。Asterisk は、PSTN と VoIP ネットワーク間のリアルタイムな接続を可能にします。Asterisk は単なる PBX をはるかに超えた存在であるため、既存の PBX を大幅にアップグレードできるだけでなく、以下のような新しい電話機能を実現できます。

- ブロードバンドインターネット経由で、在宅勤務の従業員をオフィスの PBX に接続する。
- IP ネットワーク、プライベートネットワーク、あるいはインターネット自体を介して、離れた場所にある複数のオフィスを接続する。
- Web や電子メールと統合されたボイスメールを従業員に提供する。
- 注文システムやその他のアプリケーションへの接続を可能にする IVR のようなアプリケーションを構築する。
- シンプルなブロードバンド接続や VPN 接続を使用して、どこからでも会社の PBX にアクセスできるようにする。
- その他多数……。

Asterisk には、以前はハイエンドシステムにしか搭載されていなかった以下のような高度なリソースが含まれています。

- 通話キューで待機中の顧客向けの保留音（メディアストリーミングや MP3 ファイルをサポート）。
- エージェントチームが通話に応答し、キューを監視できる通話キュー。
- テキスト読み上げ（TTS）および音声認識との統合。
- テキストファイルおよび SQL データベースの両方に転送される詳細な通話記録。
- デジタル回線およびアナログ回線を通じた PSTN 接続。

## AsteriskNOW（歴史的経緯）と FreePBX

「classic asterisk」（Debian パッケージの名称）として知られる最も純粋な形態の Asterisk は、それ自体が完成品というよりは開発ツールに近いものと見なされています。AsteriskNOW は、Asterisk をソフトアプライアンスに変えるための取り組みでした。このディストリビューションには、オペレーティングシステムとして CentOS が、グラフィカルインターフェースとして FreePBX が含まれていました。AsteriskNOW はその後、提供を終了しています。

今日、標準的なターンキー Asterisk ディストリビューションは **FreePBX**（Sangoma が保守）であり、Asterisk に Web ベースの管理 GUI とモジュールエコシステムをバンドルしたものです。FreePBX は GPL に基づいてライセンスされており、www.freepbx.org から自由にダウンロードできます。商用展開向けに、Sangoma は **FreePBX Distro**（完全な Linux イメージ）とその商用製品である **PBXact** も提供しています。

## Digium™ と Sangoma の役割

アラバマ州ハンツビルに拠点を置く Digium は、1999年の設立以来、Asterisk の作成者であり主要な開発者でした。Digium は Asterisk 開発の主要なスポンサーであることに加え、Asterisk PBX 用の電話インターフェースカードやその他のハードウェアを製造し、Switchvox（SMB 市場向け）などの商用製品を作成しました。2018年、Digium はカナダのユニファイドコミュニケーション企業である **Sangoma Technologies** に買収されました。買収以来、Sangoma は Asterisk の開発を継続的に支援し、www.asterisk.org でオープンソースプロジェクトを維持する主要なスチュワードとしての役割を果たしています。

歴史的に、Digium は Asterisk を3種類のライセンス契約で提供していました。

- General Public License (GPL) Asterisk。最も利用されているバージョンです。すべての機能が含まれており、GPL ライセンスの条件に従って自由に使用および改変できます。
- Asterisk Business Edition は、Asterisk の商用バージョンでした。一部の企業は、GPL ライセンスを使用したくない、あるいは使用できない（通常はソースコードを Asterisk と一緒に公開したくないため）という理由で、ビジネスエディションを使用していました。**注:** Asterisk Business Edition は提供を終了しました。現在、Asterisk は GPL の下でのみ配布されています。
- Asterisk OEM ライセンス。Digium が Asterisk Business Edition の小売販売を終了した後も、OEM 顧客（GPL の下で自社のソースコードを公開することなく、Asterisk 上に独自の製品を構築したい機器ベンダー）に対して、その商用エディションのライセンス供与を継続しました。

### Zapata プロジェクトと Asterisk との関係

Zapata プロジェクトは、Asterisk で使用される革新的なハードウェア設計の責任者でもあった Jim Dixon によって開発されました。このハードウェアもオープンソースであるため、どの企業でも使用でき、今日では複数のメーカーがこのアーキテクチャと互換性のあるカードを製造しています。

Zapata プロジェクトは Zaptel と呼ばれるアーキテクチャを生み出し、後に DAHDI (Digium/Asterisk Hardware Device Interface) と改称されました。このアーキテクチャの主な利点の1つは、メディアストリーミング、エコーキャンセル、トランスコーディングの処理に PC の CPU を使用できることです。対照的に、既存のカードのほとんどは、これらのタスクを実行するためにデジタル信号プロセッサ (DSP) を使用しています。専用の DSP の代わりに PC の CPU を使用することで、ボードの価格が劇的に低下します。そのため、これらのカードは、他メーカーの以前からあるインターフェースよりも大幅に安価です。その一方で、これらのカードは多くの CPU リソースを必要とします。PC の CPU を不適切に使用すると、音声品質に大きな影響を与える可能性があります。最近、Digium は G.729 および G.723 をエンコードおよびデコードするための DSP を使用したコプロセッサカードを発売し、多数のチャネルに対するスケーラビリティを向上させました。

## なぜ Asterisk なのか？

私が初めて Asterisk に触れた時のことを覚えています。通常、新しいもの、特に自分がすでに知っているものと競合するものに対する最初の反応は、それを拒絶することです！2003年に起こったのはまさにそれでした。Asterisk は、私が顧客に販売していたソリューション（4 E1 VoIP Gateway）と競合しており、私が知っていたソリューションの10分の1の価格でした。この不釣り合いな価格を見て、私は Asterisk の潜在的な落とし穴や欠点を特定するために研究を始めました。例えば、当時の PC の CPU では120の G.729 同時セッションをサポートできないことがわかりました。結局、私は Gateway ソリューションで提案を勝ち取りました。しかし、この演習を通じて、Asterisk が私の顧客基盤にとって非常に高価な問題を解決できる可能性があることを発見しました。私たちは IVR、ユニファイドメッセージング、通話録音、ダイヤラーの高額な見積もりに悩まされていましたが、適切なサイジングを行えば、CPU の問題は回避可能でした。実際、わずか3年で Asterisk は私の会社の主力製品となりました（実際、Asterisk ビジネスのためだけに別の会社を設立することにしました）。私の意見では、Asterisk は電気通信における革命であり、Web サービスにとっての Apache が IP 電話にとって意味するものと同じです。

### 極端なコスト削減

従来の PBX と Asterisk をデジタルインターフェースや電話機に関して比較すると、Asterisk はそれらの PBX よりもわずかに安価です。しかし、ボイスメール、ACD、IVR、CTI などの高度な機能を加えると、Asterisk は真価を発揮します。これらの高度な機能により、Asterisk は従来の PBX よりも大幅に安価になります。実際、Asterisk PBX をローエンドのアナログ PBX と比較するのは不公平です。なぜなら、Asterisk はローエンドのアナログシステムでは利用できない非常に多くの機能を提供しているからです。

### 電話システムの制御と独立性

顧客が Asterisk の利点として最も頻繁に挙げるものの1つは、それが提供する独立性です。今日のメーカーの中には、システムのパスワードや設定ドキュメントを顧客に提供しないところさえあります。Asterisk の「DIY（自分でやる）」アプローチにより、ユーザーは完全な自由を手に入れることができます。おまけとして、ユーザーは標準的なインターフェースにアクセスできます。

### 簡単かつ迅速な開発環境

Asterisk は、AMI および AGI インターフェースを使用して、PHP や Perl などのスクリプト言語で拡張できます。Asterisk はオープンソースであり、そのソースコードはユーザーによって変更可能です。ソースコードは主に ANSI C プログラミング言語で書かれています。

### 豊富な機能

Asterisk には、従来の PBX には見られない、あるいはオプションである機能（ボイスメール、CTI、ACD、IVR、内蔵の保留音、録音など）がいくつかあります。一部のプラットフォームでは、これらの機能のコストがプラットフォーム自体の価格を上回ります。

### 電話機上の動的コンテンツ

Asterisk は C 言語や、今日の開発環境で一般的なその他の言語を使用してプログラムされます。動的コンテンツを提供する可能性は事実上無限です。

### 柔軟で強力な dialplan

Asterisk のもう1つの画期的な点は、その強力な dialplan です。従来の PBX では、最小コストルーティング (LCR) のような単純な機能でさえ、実現不可能であるか、オプション扱いでした。Asterisk を使えば、最適なルートを選択するのは簡単でクリーンです。

### Linux 上で動作するオープンソース

Asterisk の最大の機能の1つは、そのコミュニティです。公式の Asterisk ドキュメント (docs.asterisk.org)、コミュニティが管理する VoIP-Info wiki (www.voip-info.org <http://www.voip-info.org>)、電子メール配信リスト、フォーラムなど、いくつかのリソースが利用可能です。Asterisk の採用が進むにつれて、バグは迅速に発見され、修正されます。大規模なユーザーベースと活発な開発チームにより、Asterisk は世界で最も広くテストされている PBX プラットフォームの1つであり、コードベースを安定かつ成熟したものに保つのに役立っています。

### Asterisk アーキテクチャの制限

Asterisk のいくつかの制限は、Zapata 電話設計の使用に起因しています。この設計では、Asterisk は他のプラットフォームで一般的な専用のデジタル信号プロセッサ (DSP) の代わりに、PC の CPU を使用して音声チャネルを処理します。これによりハードウェアインターフェースのコストを大幅に削減できますが、システムは PC の CPU に依存することになります。私の推奨は、Asterisk を専用マシンで実行し、ハードウェアのサイジングには保守的であることです。また、CPU を消費する過剰なブロードキャスト（ループやウイルスによって引き起こされるブロードキャストストーム）を避けるために、Asterisk を別の VLAN で使用することもできます。いくつかのベンダーから提供されている新しいインターフェースカードには、エコーキャンセル、コーデック、その他の機能を処理するための DSP が含まれるようになっており、これにより Asterisk はさらに優れたものになるでしょう。

## Asterisk PBX に対する主な反論

Asterisk の採用に対する反論を耳にすることはよくありますが、ここでそれらに対処します。

### Asterisk の市場シェアは小さすぎる

市場シェアは通常、販売された PBX の数で測定されます。これらの統計は、一般的に最大の販売代理店から取得されます。Asterisk は、販売が記録されることなくダウンロードして展開できるフリーソフトウェアであるため、これらの数値では体系的に過小評価されています。それでもなお、Asterisk は世界中で非常に大規模なインストールベースを支えており（単一サーバーのオフィス PBX から大規模なキャリアやコンタクトセンターの展開まで）、オープンソース PBX エコシステム（FreePBX などのターンキーディストリビューションを含む）の背後にある支配的なエンジンであり続けています。

### 無料なら、メーカーはどうやって生き残るのか？

実際、従来の意味でのオープンソースソフトウェアメーカーというものは存在しません。Digium は1999年から Asterisk を開発し、電話インターフェースカード、Switchvox などの商用 PBX 製品、および関連ソフトウェアの販売を通じて自らを維持してきました。2018年、Sangoma Technologies が Digium を買収しました。Sangoma は引き続き Asterisk の開発に資金を提供し、商用製品（FreePBX 商用モジュール、PBXact、Switchvox）、ハードウェア販売、および専門サービスを通じて収益を上げています。

### 技術サポートを見つけるのが難しい！

Sangoma は、パートナーエコシステムを通じて、また直接製品提供を通じて、Asterisk の商用技術サポートを提供しています。認定された専門家のグローバルネットワークが、一次サポートと専門サービスを提供します。コミュニティサポートは、www.asterisk.org の Asterisk フォーラムおよびメーリングリストを通じて活発に継続されています。

### Asterisk は200以上の extension をサポートしているか？

はい、もちろんです。適切にサイジングされた単一の Asterisk サーバーは多数の extension を処理でき、Asterisk は負荷分散とフェイルオーバーを備えた複数のサーバーにユーザーを分散させることでさらにスケールし、大規模なマルチサイト展開を可能にします。

### 「ギーク」しか Asterisk をインストールできない

FreePBX（Sangoma からスタンドアロンのディストリビューションとして入手可能）を使えば、Linux に関する知識が限られている専門家でも、中程度の複雑さの PBX をインストールおよび設定できます。GUI の助けを借りれば、わずか数時間で PBX 全体を設定することが可能です。

### サーバーが故障したらどうなるのか？

Asterisk の主な利点の1つは、フォールトトレラントなシステムで実行できる能力です。2台のサーバーを並列で実行することは比較的簡単で安価です。従来の PBX でこれを試してみてください！

### 当社はオープンソースソフトウェアを使用していない

あなたの会社は、おそらく気づかないうちにオープンソースソフトウェアを使用しています。多くのアプライアンスがオペレーティングシステムとして Linux を使用しています。さらに、Sangoma とその認定パートナーネットワークから、商用サポートや管理された展開を利用できます。

### シグナリングとメディアの処理に PC の CPU を使用することは推奨されない

Asterisk は、専用の DSP を持つ代わりに、サーバーの CPU を使用して音声チャネルのシグナリングとメディアを処理します。これにより最大5倍のコスト削減が可能になりますが、システムはメイン CPU のパフォーマンスに依存することになります。適切なサイジングを行えば、Asterisk は大容量を処理できます。それでもメイン CPU をこれらのタスクから解放したい場合は、ハードウェアエコーキャンセルや、DSP ベースの Sangoma (旧 Digium) TC400B のようなトランスコーダーカードを使用することもできます。

### チャネル

チャネルは電話回線に相当するものですが、デジタル形式です。通常、アナログまたはデジタル (TDM) シグナリングシステム、あるいはコーデックとシグナリングプロトコルの組み合わせ（例：SIP-GSM、IAX-uLaw）で構成されます。当初、すべての電話接続はアナログであり、エコーやノイズの影響を受けやすいものでした。その後、ほとんどのシステムがデジタルシステムに変換され、アナログ音声はほとんどの場合パルス符号変調 (PCM) を使用してデジタル形式に変換されました。この形式により、圧縮なしで64キロビット/秒での音声伝送が可能になります。

公衆交換電話網 (PSTN) とインターフェースするチャネル：

- `chan_dahdi`: Sangoma (旧 Digium)、Xorcom 等によるアナログ (FXO/FXS) およびデジタル (E1/T1/PRI) TDM カード。DAHDI に対して個別にビルドされます — *Legacy channels* の章を参照してください。

Voice over IP とインターフェースするチャネル：

- `chan_pjsip`: SIP — Asterisk 22 LTS における唯一の主要な SIP チャネルドライバー。ダイヤル文字列: `PJSIP/endpoint_name`。(**注:** 古い `chan_sip` は Asterisk 21 で削除され、Asterisk 22 には存在しません。設定については *Building your first PBX with PJSIP* を参照してください。)
- `chan_iax2`: IAX2 プロトコル — Asterisk 22 にも同梱されていますが、レガシーです。新規展開には SIP/PJSIP が推奨されます。ダイヤル文字列: `IAX2/peer`。
- `chan_unistim`: Nortel/Avaya UNISTIM 電話機。現在も利用可能（延長サポート）ですが、めったに使用されません。

古い VoIP チャネルは、標準の Asterisk 22 ビルドの一部ではなくなりました：`chan_h323` (H.323) はコミュニティの `ooh323` アドオンとしてのみ存続しており、`chan_mgcp` (MGCP) および `chan_skinny` (Cisco SCCP) は非推奨となり、最新のチャネルセットから削除されました。これらのプロトコルと相互運用する必要がある場合は、Asterisk の前にゲートウェイを配置するのが一般的なアプローチです。

その他のチャネル：

- **Local**: 異なる context の dialplan にループバックする擬似チャネル（コアに組み込まれています） — 再帰的なルーティングや、通話を複数の宛先に分岐させるのに便利です。ダイヤル文字列: `Local/extension@context`。

### コーデックとコーデック変換

通常、データネットワーク上で可能な限り多くの音声接続を収容しようとします。コーデックはデジタル音声における新しい機能を可能にし、その中には圧縮が含まれます。これは、8対1を超える圧縮率を可能にする最も重要な機能の1つです。多くのコーデックは、音声アクティビティ検出（無音抑制）、パケット損失隠蔽、コンフォートノイズ生成などの機能も定義していますが、Asterisk 自体はコンフォートノイズを生成したり、無音抑制を実行したりはしません。Asterisk ではいくつかのコーデックが利用可能であり、それらの間で透過的に変換できます。内部的には、Asterisk はあるコーデックから別のコーデックに変換する必要がある場合、ストリーム形式として slinear を使用します。Asterisk の一部のコーデックはパススルーモードでのみサポートされており、これらのコーデックは変換できません。システムにインストールされているコーデックを確認するには、コンソールコマンドを使用できます：

```
CLI>core show translation
```

以下のコーデックがサポートされています：

- G.711 ulaw (USA) - (64 Kbps)
- G.711 alaw (Europe) - (64 Kbps)
- G.722 (High Definition) – (64 Kbps)
- G.723.1 - パススルーモードのみ
- G.726 - (16/24/32/40kbps)
- G.729 - Sangoma が配布するバイナリコーデックモジュール。ダウンロードは無料ですが、合法的に使用するにはチャネルごとのライセンス購入が必要です (8Kbps)
- GSM - (12-13 Kbps)
- iLBC - (15 Kbps)
- LPC10 - (2.4 Kbps)
- Speex - (2.15-44.2 Kbps)
- Opus - (6-510 Kbps)

### プロトコル

ある電話機から別の電話機へデータを送信することは、データが自力で相手の電話機へのパスを見つけることができれば簡単であるはずです。残念ながらそうはならず、電話機間の接続を確立し、エンドデバイスを発見し、電話シグナリングを実装するためにシグナリングプロトコルが必要です。SIP は現代の展開における支配的なシグナリングプロトコルであり、Asterisk 22 LTS で利用可能な唯一の SIP チャネルです（chan_pjsip 経由）。IAX2 は依然として利用可能ですが、レガシーと見なされています。Asterisk は以下のプロトコルをサポートしています。

- SIP — `chan_pjsip` 経由
- IAX2 — レガシー、Asterisk 22 に同梱
- UNISTIM — Nortel/Avaya 電話機（延長サポート）
- H.323, MGCP, および SCCP (Cisco Skinny) — 標準の Asterisk 22 ビルドには含まれないレガシープロトコル（H.323 はコミュニティの `ooh323` アドオン経由のみ）

### アプリケーション

ある電話機から別の電話機へ通話をブリッジするには、アプリケーション dial() が使用されます。ほとんどの Asterisk 機能（ボイスメールや会議など）はアプリケーションとして実装されています。利用可能な Asterisk アプリケーションは、core show applications コンソールコマンドを使用して確認できます。

```
CLI>core show applications
```

Asterisk アドオン、サードパーティプロバイダー、あるいは自分で開発したアプリケーションを追加することもできます。

## Asterisk システムの概要

Asterisk は、TDM や IP 電話などの技術を統合し、ハイブリッド PBX として機能するオープンソース PBX です。Asterisk は、対話型音声応答 (IVR) や自動着信分配 (ACD) などの機能を実装する準備ができており、前述のように、新しいアプリケーションの開発にも開かれています。この図は、Asterisk がアナログおよびデジタルインターフェースを使用して PSTN や既存の PBX にどのように接続するか、またアナログ電話や IP 電話をどのようにサポートするかを示しています。ソフトスイッチ、メディアゲートウェイ、ボイスメール、音声会議として機能し、保留音も内蔵しています。

## テストシステムの構築

Asterisk ソリューションを実装する際、最初のステップは通常、テストマシンを構築することです。最も簡単なテストマシンは、少なくとも1台の電話機と1回線を含む 1x1 PBX です。これを行うにはいくつかの方法があります。

![シンプルな Asterisk テストシステム](../images/01-introduction-fig05.png)

### 1つの FXO、1つの FXS

テストマシンを構築する最初で最も簡単な方法は、1つの FXO インターフェースと1つの FXS インターフェースを備えたカードを購入することです。FXO ポートを既存の回線に接続し、FXS をアナログ電話に接続します。これで 1x1 PBX が完成します。

### VoIP サービスプロバイダー：ATA

これは VoIP オプションです。この場合、音声サービスプロバイダーに登録して SIP trunk を取得し、SIP アナログ電話アダプターを購入する必要があります。PC をすでにお持ちであれば、100ドル未満で済むでしょう。

### 安価な FXO カードまたは ATA

私は安価な FXO カードから始めました。一部の安価な V.90 FAX モデムは、FXO カードとして Asterisk で動作します。初期の Digium カードの一部は、これらのカードを使用して作成されました（例：X100P および X101P）。これらは Motorola および Intel チップセット（Motorola 68202-51、Intel 537PU、Intel 537PG、Intel Ambient MD3200 が動作することが知られています）に基づいた古いモデムです。これらのモデムは、新しいマザーボードと互換性がないことがよくあります。最近、一部のメーカーがこれらのカードを X100P クローンとして販売し始めました。互換性の問題の一部はパッチを使用して解決できます。詳細については以下を参照してください：

- http://www.voip.school/mediawiki/index.php/Asterisk_patch_for_the_X100P_card

## Asterisk のシナリオ

Asterisk はいくつかの異なるシナリオで使用できます。その一部を挙げ、それぞれの利点と制限の可能性を説明します。

### IP PBX

最も一般的なシナリオは、新しい PBX のインストール、または既存の PBX の置き換えです。Asterisk を他の代替手段と比較すると、現在市場で入手可能なほとんどの PBX よりも安価で機能が豊富であることがわかります。現在、多くの企業が他のブランド名 PBX ではなく Asterisk に仕様を変更しています。

![IP PBX としての Asterisk](../images/01-introduction-fig06.png)

### レガシー PBX の IP 化

次の画像は、最も一般的に使用されるセットアップの1つを示しています。大企業は通常、新しい技術に投資する際に大きなリスクを負いたがらず、同時にレガシー機器への投資を維持したいと考えています。レガシー PBX の IP 化は非常に高価になる可能性があるため、T1/E1 回線を使用して Asterisk PBX を接続することは、コストを意識する顧客にとって良い代替案となります。もう1つの利点は、より良い通話料金で VoIP サービスプロバイダーに接続できる可能性があることです。

![レガシー PBX の IP 化](../images/01-introduction-fig07.png)

### Toll Bypass

VoIP の非常に便利なアプリケーションは、インターネットや WAN を介して支店を接続することです。既存のデータ接続を使用することで、本社と支店間の通信接続で発生する通話料金を回避できます。

![WAN を介したオフィス間の Toll bypass](../images/01-introduction-fig08.png)

### アプリケーションサーバー（IVR、会議、ボイスメール）

Asterisk は、既存の PBX のアプリケーションサーバーとして使用したり、PSTN に直接接続したりできます。Asterisk は、ボイスメール、FAX 受信、通話録音、データベースに接続された IVR、音声会議サーバーなどのサービスを提供します。ボイスメールと FAX を既存の電子メールサーバーに統合すれば、ユニファイドメッセージングシステムが構築できますが、これは通常高価なソリューションです。Asterisk をアプリケーションサーバーとして使用すると、他のソリューションと比較して極端なコスト削減が実現します。

![アプリケーションサーバーとしての Asterisk](../images/01-introduction-fig09.png)

### メディアゲートウェイ

ほとんどの Voice over IP サービスプロバイダーは、SIP ユーザーのすべての登録、場所、認証をホストするために SIP プロキシを使用しています。彼らは依然として、SIP または H.323 Voice over IP 接続を使用して、通話を直接 PSTN に送信するか、卸売通話終了プロバイダーを通じてルーティングする必要があります。Asterisk はバック・ツー・バック・ユーザーエージェント (B2BUA) またはメディアゲートウェイとして機能し、非常に高価なソフトスイッチやメディアゲートウェイを置き換えることができます。主要な市場メーカーの4 E1/T1 ゲートウェイの価格と Asterisk を比較してください。Asterisk ソリューションは他のソリューションよりも数倍安価であり、シグナリングプロトコル（H.323、SIP、IAX…）やコーデック（G.711、G.729…）を変換できます。

![メディアゲートウェイとしての Asterisk](../images/01-introduction-fig10.png)

### コンタクトセンタープラットフォーム

コンタクトセンターは、自動着信分配 (ACD)、対話型音声応答 (IVR)、通話監視など、いくつかの技術を組み合わせた非常に複雑なソリューションです。基本的に、インバウンド、アウトバウンド、ブレンデッドの3種類のコンタクトセンターがあります。インバウンドコンタクトセンターは非常に洗練されており、通常は ACD、IVR、CTI、録音、監視、レポートが必要です。Asterisk には、通話をキューに入れるための ACD が組み込まれています。IVR は Asterisk Gateway Interface (AGI) またはアプリケーション background() などの内部メカニズムを使用して実行できます。コンピュータ電話統合 (CTI) は Asterisk Manager Interface (AMI) を使用して実現され、録音とレポートは Asterisk に組み込まれています。アウトバウンドコンタクトセンターの場合、予測ダイヤラーまたはパワーダイヤラーが主要なコンポーネントの1つです。オープンソースの Asterisk 用にいくつかのダイヤラーが利用可能ですが、必要であればプラットフォーム用に独自のダイヤラーを構築することも難しくありません。ブレンデッドコンタクトセンターは、インバウンドとアウトバウンドの同時運用を可能にし、エージェントの時間をより有効に活用することでコストを節約します。Asterisk とその ACD メカニズムを使用して、ブレンデッドソリューションを実装することが可能です。

![Asterisk コンタクトセンタープラットフォーム](../images/01-introduction-fig11.png)

## まとめ

Asterisk は GPL に基づいてライセンスされたソフトウェアであり、一般的な PC を強力な IP PBX プラットフォームとして機能させることができます。Digium の Mark Spencer が1990年代後半に Asterisk を作成し、Digium は Asterisk 関連のハードウェアと商用製品を販売することで自らを維持してきました。Digium は2018年に Sangoma Technologies に買収されました。現在、Sangoma が Asterisk の開発を支援しています。ハードウェアインターフェースの設計は Jim Dixon が開発した Zapata プロジェクトに端を発し、それが DAHDI を生み出しました。

Asterisk アーキテクチャには、以下の主要なコンポーネントがあります：

- CHANNELS: アナログ、デジタル、または Voice over IP。Asterisk 22 LTS では、SIP は chan_pjsip によってのみ処理されます。
- PROTOCOLS: 通話のシグナリングを担当する通信プロトコル。SIP (PJSIP 経由)、H323、MGCP、IAX2 などが含まれます。
- CODECS: 音声のデジタル形式を変換し、圧縮やパケット損失隠蔽を可能にします。Asterisk 自体は無音抑制（音声アクティビティ検出）やコンフォートノイズ生成を行わないことに注意してください。エンドポイントが VAD を使用する場合、クライアント側でコンフォートノイズを無効にする必要があります。
- APPLICATIONS: Asterisk PBX の機能性を担当します。会議、ボイスメール、FAX は Asterisk アプリケーションの例です。

Asterisk は、小規模な IP PBX から洗練されたコンタクトセンターまで、さまざまなシナリオで使用できます。www.asterisk.org および docs.asterisk.org で簡単にヘルプを見つけることができます。
